# Методология

← [README](../README.md) · [конфигурация](configuration.md) · [установка](install.md)

Probe-вердикт ladon — это не один сигнал, а композиция четырёх независимых
наблюдений + три слоя arbitration над ними. Этот документ описывает каждый слой

## Что значит вердикт

Решение по домену сводится к одному из двух действий:

- **в туннель** — IP-адреса домена добавляются в kernel ipset
  `ladon_engine`, iptables направляет соответствующий трафик в VPN.
  Свежее решение помечается как `Hot` — TTL 24h, домен периодически
  перепробуется. Если за окно 24h набирается ≥N подтверждённых failures
  (default N=50, ≈4h устойчивого failure'а при 5-минутном cooldown) —
  переходит в `cache`, постоянный туннелируемый список без TTL.
- **direct** — ladon ничего не делает, домен ходит обычным маршрутом.
  В терминах state machine — `Ignore`.

Дальше по тексту используются эти короткие имена: `Hot` / `cache` —
домен в туннеле, `Ignore` — direct.

## Источник наблюдений — реактивная подписка

Анализ применяется только к доменам, к которым клиенты **уже**
обратились. Probe-кандидат — это имя из живого DNS-трафика, не из
синтетического списка известных блокировок. Покрытие = реальный профиль
использования сети, не публикуемый "топ заблокированных".

## Probe как структурированное измерение

Один probe = последовательная серия из 4 независимых наблюдений:

1. **DNS** — резолвится ли имя
2. **TCP:443** — пускают ли SYN, приходит ли SYN/ACK
3. **TLS handshake** — проходит ли ClientHello/ServerHello, доходит до Finished
4. **HTTP read** — приходит ли первые ~32KB ответа после успешного handshake

Каждое наблюдение даёт `passed`/`failed` + при failure — категорию
ошибки. Категория несёт информацию о **природе** отказа, не только о
факте.

## Семантика failures: server-active vs path-active

Failures разделяются по природе на два класса.

**Server-active rejection** — сервер активно ответил "нет":
- TCP `connection refused` — порт явно закрыт
- любой TLS alert от peer'а — `certificate required` (mTLS),
  `illegal parameter`, `handshake failure`, и т. д.

Эти отказы **доказывают**, что сервер достижим — чтобы прислать TLS
alert или RST на `connect()`, peer должен был получить и обработать наш
пакет. DPI на пути не подделывает TLS alert'ы (это требовало бы
поддерживать TLS state machine на L6), поэтому такие сигналы
интерпретируются как server-side policy, не как блок. Классифицируется
сразу в `Ignore`.

**Path-active rejection** — сервер не подтвердил доступность из этой точки:
- timeout (любой стадии)
- TCP/TLS reset
- silent EOF (TCP закрылся без HTTP/TLS-ответа)
- HTTP cutoff (стрим обрублен mid-response)
- HTTP-level garbage (вместо валидного HTTP пришёл бинарный мусор —
  типичная сигнатура DPI, инжектящего фальшивый response в HTTP/2 stream)

Это и есть рабочий сигнал блокировки в local-режиме: с точки зрения
пользователя домен **не работает**, и default-действие — отправить его в
Hot, чтобы трафик пошёл через тоннель — корректное. С одной точки
наблюдения нельзя отличить DPI от других причин с тем же симптомом
(сервер прилёг, geoblock на всю РФ, anti-bot отбил probe), но действие
от этого не меняется: маршрутизация через тоннель либо помогает (DPI,
geoblock), либо безвредна (мёртвый сервер всё равно мёртвый и через
тоннель). Если важно дополнительно фильтровать домены, где «не
работает» относится к серверу, а не к пути — сверху навешивается
exit-compare (см. ниже).

## Arbitration через второй observer (exit-compare)

**Опциональный** второй уровень точности. В local-режиме path-active
failure → Hot работает корректно (см. выше); exit-compare нужен когда
оператор хочет дополнительно отсекать класс «домен не работает не из-за
DPI, а потому что у самого сервера/CDN проблема» — тащить такие домены в
тоннель бессмысленно (через тоннель тоже не заработает) и засоряет
ipset.

Идея простая — **вторая независимая точка наблюдения**: тот же probe из
другой географической позиции (out-of-region vantage point — чужая VPS,
другой ISP, другой регион). Получаем матрицу 2×2:

|  | remote OK | remote FAIL |
|---|---|---|
| **local OK** | `Ignore` (direct работает) | (вырожденный, не вызываем remote) |
| **local FAIL** | **`Hot`** (снаружи живой → блок именно у нас) | **`Ignore`** (отказ из обеих точек → server-side, не путь) |

Ключевая строка — `local FAIL + remote FAIL → Ignore`. Это и есть тот
класс FP, который exit-compare закрывает: dead/dying сервер, anti-bot
endpoint, который отбивает любой probe независимо от вантажа,
geoblock на широкую географию (включая remote-вантаж). Без exit-compare
такие домены попадают в Hot — поведение не «сломанное», но шумное.

Сравнение **постадийное** по коду failure'а: identical-stage failure из
обеих точек (типичный `http_cutoff` на anti-bot / rate-limit / geo-WAF
endpoint'ах вроде backend'ов Yandex Music, Apple-mTLS cohort'е,
mtalk.google.com) → server-side, не path-active.

Если remote-наблюдатель транспортно недоступен (timeout, network) —
сохраняется local-вердикт без overrule (sticky local). Это защита от
ситуации "remote умер → внезапно весь Hot список развалился", и заодно
гарантирует что в periods когда exit-compare выключен/недоступен ladon
работает строго как в local-режиме.

Контракт probe-сервера и поведение в exit-compare режиме — в
[probe-api.md](probe-api.md).

## Arbitration через накопление (temporal evidence)

Одна проба — слабое доказательство. Сервер мог быть перегружен в
конкретный момент, DPI мог моргнуть, провайдер мог временно
ребутнуть промежуточное оборудование. Поэтому над probe-вердиктом
строится статистический слой:

- Первый зафиксированный path-active failure → домен попадает в Hot
  на 24 часа (активный, перепробуется)
- Если за 24h собирается ≥N подтверждённых failures (при дефолтных
  настройках N=50, что при 5-минутном cooldown означает минимум ~4
  часа устойчивого failure'а) → блок считается стабильным, переходит в
  постоянный список без TTL
- Если за 24h не собирается — TTL истекает, домен возвращается в обычный
  probe-цикл

Случайные блипы провайдера и временные DPI-моргания не доходят до
постоянного списка.

## Сводная декомпозиция

Классификация домена выводится из независимых сигналов:

1. **stage** — где именно проба упала
2. **природа failure** — server-active или path-active (закрывает класс
   FP — mTLS / cert-only сервисы и любой ответивший RST/alert)
3. **накопление во времени** — статистический threshold на устойчивости
   (закрывает класс — случайные блипы провайдера и моргания DPI)
4. **(опционально) второй observer** — exit-compare matrix (закрывает
   класс — server-side проблемы и широкий geoblock, где «не работает»
   относится к серверу, а не к пути)

Слои 1-3 работают всегда; слой 4 — когда `probe.mode: exit-compare`.
В local-режиме домен в постоянном списке прошёл фильтр на трёх уровнях,
в exit-compare — на четырёх.

## Чего методология не различает

1. **L7-fingerprint discrimination** — DPI блокирует только конкретный
   ClientHello (например, Chrome) и пропускает другие. Probe — отдельный
   TLS-клиент с собственным fingerprint'ом; если DPI его не блеклистит,
   probe видит "OK", а реальный браузер юзера — нет. Без mimicry
   ClientHello не детектируется автоматически.

2. **Throttling vs blocking** — DPI шейпит bandwidth до низких значений
   вместо обрыва. Probe видит "медленно, но работает" → `Ignore`. Для
   детекции нужна отдельная проба на sustained throughput, не
   покрывается базовой методологией.

3. **Domain-less flows** — TCP/UDP-соединения на hardcoded IP без
   предварительного DNS-запроса (Telegram mobile DC, Discord voice,
   WhatsApp calls, Steam P2P). Реактивная подписка работает только на
   DNS-наблюдении: нет наблюдения — нет probe-кандидата. Известные
   CIDR-диапазоны оператор может пре-загрузить через `extensions`
   (см. `telegram` MTProto-блоки) — попадают в `ladon_cidr` напрямую,
   минуя probe.

4. **DNS-only blocks** — DPI poison'ит ISP-resolver, отдавая клиенту
   неверные IP. Probe резолвит через альтернативный апстрим (off-ISP
   DNS) → видит правильный IP → не замечает фальсификации, которую
   видит клиент через ISP-DNS. Для детекции нужна параллельная проба
   через ISP-resolver и сравнение результата.
